ШАГ 2. Обнови генерацию USER PROMPT.
Склеивай user prompt перед отправкой в LLM так:
1. Начни с: "Оцени кандидата по следующим правилам:\n".
2. Добавь из CRITERIA_RULES тексты ТОЛЬКО тех критериев, которые выбраны чекбоксами.
3. Если включен счетчик филлеров, добавь FILLER_WORDS_RULE.
4. Если оценивается 1 кейс: добавь "ТРАНСКРИПТ ДИАЛОГА:\n[текст]".
5. Если оценивается 2 кейса: добавь TWO_CASES_RULE, затем "ТРАНСКРИПТ КЕЙСА 1:\n[текст1]\n\nТРАНСКРИПТ КЕЙСА 2:\n[текст2]".


ШАГ 3. Напиши агрегацию оценок (на стороне бэкенда после получения JSON).
Если в ответе LLM есть и "case_1", и "case_2", примени правило наихудшего сценария:
Для каждого оцененного критерия найди минимальный балл. Пример: min_score = min(case_1.scores[key], case_2.scores[key]).
Используй эти минимальные баллы для расчета итогов. Если кейс один — бери баллы из "case_1".


ШАГ 4. Напиши расчет итогового вердикта (на бэкенде).
1. Установи лимит филлеров: если кейс один = порог из UI. Если кейсов два = порог из UI умножить на 2.
2. ЖЕСТКИЕ СТОПЫ. Вердикт = "Не рекомендуется", если выполняется ХОТЯ БЫ ОДНО:
   - "filler_words_count" СТРОГО БОЛЬШЕ лимита.
   - Хотя бы один минимальный балл в scores равен 0.
3. Если стопов нет, считай процент: P = (Сумма минимальных баллов / (Количество ключей в scores * 2)) * 100.
4. Вердикт по процентам:
   - P >= 75: "Рекомендуется"
   - P >= 40 и P < 75: "Требуется дополнительная проверка"
   - P < 40: "Не рекомендуется"
